อธิบาย Scenarios

การทำงานของแต่ละสถานการณ์จำลอง

⚡ Scenario 1 — Local Cache Hit (ดึงข้อมูลจากแคชภายใน)

สถานการณ์: Node "Ben-01" ต้องการ DNA ของ Tetramand ซึ่งมีอยู่ในแคชของตัวเองอยู่แล้ว

การทำงาน: ระบบตรวจสอบแคชภายในของ Node แล้วพบข้อมูลที่ต้องการ → ดำเนินการ Transformation ได้ทันที โดยไม่ต้องใช้เครือข่ายเลย ทำให้ Latency ต่ำมาก (~1-5 ms)

สิ่งที่พิสูจน์: ระบบ Tier 1 (Local Authority) สามารถทำงานได้แม้ไม่มีการเชื่อมต่อเครือข่าย — เหมาะสำหรับสถานการณ์ฉุกเฉินที่ต้องการ real-time response

🤝 Scenario 2 — Federated Consensus (ลงมติแบบกระจายศูนย์)

สถานการณ์: Node "Ben-01" ต้องการ DNA ของ Galvan ซึ่งไม่มีในแคช (เอเลี่ยนตัวใหม่ที่ยังไม่เคยใช้)

การทำงาน: Node ส่ง Interest Packet ไปยัง Plumber Nodes ทั้ง 3 ตัว → Plumber Nodes รันกระบวนการ PBFT Consensus (ลงมติร่วมกัน) เพื่อตรวจสอบว่าคำขอนี้ผ่านข้อกำหนดด้านจริยธรรม → ถ้าผ่าน ก็จะออก Capability Token และส่ง Data Packet กลับมาให้

สิ่งที่พิสูจน์: ระบบ Tier 2 (Federated Consensus) ใช้การลงมติแบบกระจายศูนย์เพื่อปลดล็อกข้อมูลใหม่ — Latency สูงกว่า Tier 1 (~150-350 ms) แต่มีความปลอดภัยสูงเพราะต้องผ่านการตรวจสอบจากหลาย Node

💔 Scenario 3 — Network Partition (เครือข่ายถูกแบ่งแยก)

สถานการณ์: Node "Gwen-02" ถูกตัดขาดจากเครือข่าย (เช่น สัญญาณขาดหาย, link failure) แล้วทดสอบ 2 กรณี

การทำงาน:
• Test A: ขอ DNA ที่มีในแคช (Tetramand) → ✅ สำเร็จ! ใช้ข้อมูลจากแคชได้แม้ไม่มีเน็ต
• Test B: ขอ DNA ที่ไม่มีในแคช (Kineceleran) → ❌ ล้มเหลว! ต้องใช้ Consensus แต่ติดต่อ Plumber Nodes ไม่ได้

สิ่งที่พิสูจน์: สถาปัตยกรรม ONet ทนต่อ Network Partition ได้ — ข้อมูลสำคัญที่แคชไว้แล้ว ยังคงใช้งานได้เสมอ (life-critical safety) แม้เครือข่ายจะล่ม

📦 Scenario 4 — In-Network Caching (แชร์ข้อมูลผ่าน Peer Cache)

สถานการณ์: Ben-01 ดึง DNA ของ Galvan จาก Server แล้ว Cache ไว้ → จากนั้น Gwen-02 ขอ DNA ตัวเดียวกัน

การทำงาน: เมื่อ Gwen-02 ส่ง Interest Packet ออกไป ระบบ NDN จะค้นหาข้อมูลจาก Node ที่ใกล้ที่สุดที่มีข้อมูลนั้นอยู่ → พบว่า Ben-01 มีอยู่ในแคชแล้ว → ส่ง Data จาก Ben โดยตรง โดยไม่ต้องไปถึง Server ทำให้ Latency ลดลงมาก (~10-30 ms)

สิ่งที่พิสูจน์: ระบบ In-Network Caching ของ NDN ช่วยให้ Node ต่างๆ แชร์ข้อมูลกันได้แบบ peer-to-peer — ลดภาระจาก Server กลาง และเพิ่ม fault tolerance

🚫 Scenario 5 — Consensus Rejection (ถูกปฏิเสธจากระบบ Consensus)

สถานการณ์: Ben-01 ขอ DNA ของ Ectonurite — สิ่งมีชีวิตที่ถูกจำกัดการเข้าถึง ภายใต้ระบบ Alien Species Preservation System (ASPS)

การทำงาน: Plumber Nodes ทั้ง 3 ตัวรับคำขอ → รัน PBFT Consensus → ตรวจสอบกับนโยบาย ASPS → พบว่า Ectonurite เป็นสิ่งมีชีวิตที่ต้องอนุรักษ์ → ทุก Node โหวต REJECT ❌ → ส่ง Reject Packet กลับไป → Ben-01 ไม่ได้รับ Capability Token

สิ่งที่พิสูจน์: ระบบ Federated Consensus ป้องกันการเข้าถึงข้อมูลที่ไม่ได้รับอนุญาต — ทำหน้าที่เป็น access control แบบกระจายศูนย์ที่ไม่มีจุดล้มเหลวจุดเดียว (no single point of failure)

🏃 Scenario 6 — Node Mobility (Node เคลื่อนที่ในเครือข่าย)

สถานการณ์: Ben-01 เป็น Node ที่เคลื่อนที่อยู่ตลอดเวลา — เริ่มจากอยู่ใกล้ Plumber-1 แล้วเคลื่อนไปอยู่ใกล้ Plumber-2

การทำงาน: Ben-01 ดึง DNA จาก Plumber-1 ตอนอยู่ใกล้ → จากนั้นเคลื่อนที่ข้ามเครือข่าย → Topology อัพเดตแบบ dynamic (links เปลี่ยนตามตำแหน่ง) → เมื่อขอ DNA ใหม่ ระบบจะ route ไปยัง Plumber-2 ที่อยู่ใกล้ที่สุดโดยอัตโนมัติ — ไม่มี session หลุด

สิ่งที่พิสูจน์: การ routing แบบ NDN ปรับตัวตาม Node mobility ได้ — ไม่เหมือน TCP/IP ที่ต้อง re-establish connection ใหม่เมื่อเปลี่ยนเครือข่าย (seamless handoff)

🔀 Scenario 7 — Multi-hop Routing (ส่งข้อมูลผ่านหลาย Node)

สถานการณ์: Gwen-02 ต้องการ DNA ของ Wildmutt แต่ไม่มี link ตรงไปยัง Plumber-1 ที่เก็บข้อมูลนั้น — ต้อง relay ผ่าน Plumber-3

การทำงาน:
• Hop 1: Gwen-02 ส่ง Interest → Plumber-3 (ไม่มีข้อมูล → forward ต่อ)
• Hop 2: Plumber-3 ส่ง Interest → Plumber-1 (พบข้อมูล!)
• กลับ: Plumber-1 ส่ง Data → Plumber-3 (cache ไว้ระหว่างทาง) → Gwen-02

สิ่งที่พิสูจน์: NDN routing ส่งข้อมูลแบบ multi-hop ได้โดยไม่ต้องมี direct connection — และ Node ที่อยู่ระหว่างทาง (Plumber-3) จะ cache ข้อมูลไว้อัตโนมัติ เพื่อให้การขอข้อมูลครั้งถัดไปเร็วขึ้น
